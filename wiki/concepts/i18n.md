# Bilingual (DE/EN) duplicated-tree i18n

## Summary

Shared/homepage strings swap client-side via `data-i18n`; the blog is mirrored as separate DE/EN
file trees with URL redirects. After the Blackwall redesign the in-page mechanism grew a
`data-i18n-attr` channel and an `onLanguageChange` re-render hook.

## Details

Two complementary mechanisms:

1. **For shared/homepage UI**: [[i18n-system]] (`js/translations.js`) holds a DE+EN dictionary
   (now including the ~70-key `bw.*` Blackwall namespace) and `setLanguage()` rewrites
   `[data-i18n]` in place ‚Äî as `textContent`, as `innerHTML` (`data-i18n-html`, for copy that
   contains markup like the `wAIser` wordmark spans and `.rip` highlights), or as a named attribute
   (`data-i18n-attr`, e.g. the scan input `placeholder`). It persists to `localStorage`, flips
   `<html lang>`, and then fires `window.onLanguageChange(lang)` so **JS-rendered content**
   (blog feed, blog list, article "related" grid, living-button labels ‚Äî all owned by
   [[main-interactions]] / `js/site.js`) re-renders in the new language rather than going stale.
2. **For long-form content**: translation is *structural* ‚Äî `blog/` + `blog/posts/de/` vs
   `blog/en/` + `blog/posts/en/`, each post a separately authored file (16 per language; see
   [[blog-system]]). Toggling language on a blog URL **redirects** to the mirror path, with a
   `slugMap` covering the **four** posts whose filenames differ across languages
   (`mit-ai-halluzinationen` ‚Üî `mit-ai-hallucinations`,
   `ambient-ai-die-naechste-ki-generation` ‚Üî `ambient-ai-the-next-ai-generation`,
   `loops-statt-prompts-cherny` ‚Üî `loops-not-prompts-cherny`,
   `opus-4-8-dynamische-workflows-erst-recht-audit` ‚Üî `opus-4-8-dynamic-workflows-erst-recht-audit`).

German is the default/canonical language (`lang="de"`; the homepage is a single bilingual URL with
`hreflang="x-default"` only). EN is detected from the URL for blog pages and from `localStorage`
elsewhere. Per-post hreflang tags and the sitemap encode the de/en/x-default relationships for SEO
(see [[seo-geo-layer]]).

This structural approach is a direct consequence of the [[static-no-build]] constraint ‚Äî there is
no server-side render step to produce two language variants from one source, so the post files are
physically duplicated while the shared chrome and homepage swap client-side.
